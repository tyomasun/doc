# GitLab MR review workflow

Установка и конфиг:
- Значение сохраняется в `local_config.json` внутри каталога скилла.
- Переопределение при запуске: env-переменная `GIT_REPO_ROOTS` имеет приоритет над `local_config.json`.

Когда пользователь даёт ссылку на GitLab Merge Request (MR), сделай:

1) Запусти подготовительный скрипт (не из проекта, а из каталога навыков пользователя):
   - powershell -ExecutionPolicy Bypass -File "<skill_root>\\run_mr_prep.ps1" "<MR_URL>"
      (где <skill_root> — каталог текущего навыка)

2) Скрипт должен вывести JSON с полями:
   - repo_dir, project_path, target_branch, source_branch
   - stubs_path (путь к стабам из `SBIS_STUBS_PATH`, резервно можно передать через `SABY_STUBS_PATH`)
   - diff_unified_path (файл с unified diff)
   - changes_json_path (файл с JSON changes от GitLab API)
   - notes (если были эвристики/предупреждения)
   - при этом команды git печатаются в консоль (stderr), JSON остаётся в stdout

3) Перейди в `repo_dir`, убедись что целевая ветка обновлена (скрипт это уже делает), и используй `diff_unified_path` + `changes_json_path` как источники для ревью.
4) Обязательно обработай ВЕСЬ diff:
   - Прочитай `changes_json_path` и выпиши полный список файлов.
   - Просмотри весь `diff_unified_path` целиком и убедись, что каждый файл из списка имеет просмотренные hunks.
   - Если diff большой, читай по частям, но не пропускай секции и не ограничивайся первыми фрагментами.
5) Для проверки контрактов бери путь к стабам из поля `stubs_path`, которое вернул `mr_prep.py`.
   - Если `stubs_path` пустой, отсутствует на диске или указан только в `notes` как проблемный, зафиксируй это как ограничение ревью и не подставляй захардкоженный путь.
   - Путь настраивается через env `SBIS_STUBS_PATH` (резервно поддерживается `SABY_STUBS_PATH`).
   - Перед проверкой контрактов сделай классификацию методов:
   - Все методы, объявленные в `.orx` как `<select ... name="Module.Method">`, считай ПРИКЛАДНЫМИ API.
   - Никогда не ищи имена прикладных методов (`Module.Method`) в каталоге стабов из `stubs_path`.
   - По стабам проверяй только платформенные вызовы `sbis.*` (например `sbis.Record`, `sbis.RecordSet`), если они реально вызываются в Python-коде или в `<body>` метода.
6) Если в diff меняется базовый класс, общий метод, миксин или helper-функция, проверь не только файлы из MR, но и локальные места переиспользования:
   - Найди наследников изменённого класса и все call sites изменённого метода/функции хотя бы поиском по репозиторию.
   - Не ограничивайся каталогом diff: проверь внешние модули/пакеты, которые импортируют этот класс или вызывают этот метод.
   - Если логика зависит от нового аргумента, поля или состояния, явно проверь кто и как его передаёт у наследников и внешних вызовов.
   - Если нашёл важный внешний call site вне diff, учитывай его в выводах ревью и не формулируй замечание так, будто вызовов больше нет.

Правила ревью (после подготовки):
- Для python кода ориентируйся на Python 3.11
- Для SQL кода ориентируйся на Postgresql 10
- Используй заглушки из `stubs_path` как источник только платформенных контрактов `sbis.*`, сигнатур функций, форм данных и типов API 
- Не ищи в стабах методы прикладных модулей, объявленные в `.orx` (их там нет по определению). Прикладными считаются вызовы вида: sbis.BLObject.BLMethod() или BLObject.BLMethod(). Платформенными считаются вызовы вида: sbis.Record() или Record()
- Ищи: баги, гонки, NPE/KeyError, неправильные условия, утечки ресурсов, ошибки обработки ошибок/ретраев, нарушения API контрактов.
- Ошибки импорта
- Ищи security: секреты в коде, небезопасная сериализация, shell injection, path traversal, отключение TLS verify, слабая authz.
- Ищи качество: отсутствие тестов для критичных изменений, нарушения стиля/архитектуры, неочевидная логика без комментариев.
- Доп.проверка для Saby/CRM CRUD:
  - Если в MR меняются `.orx`, `.trg`, `before_*`/`after_*` обработчики, проверь lifecycle автогенерируемого CRUD-метода и session-контекстов (`set_config/current_setting`) по `.orx` + коду обработчиков.
  - Для детальной платформенной логики опирайся на `C:/Users/an.kochetovskiy/.codex/skills/sabyapp/references/crud-autogen-review.md`.
- Опиши суть и логику изменений
- Являются ли изменения закноченными или есть явные недоделки
- Можно ли сделать лучше? - Предложи если возможно оптимизацию
- Возвращай результат списком замечаний: [severity][file:line] проблема → рекомендация/патч.
- Для файлов, которые существуют в текущем локальном checkout, используй markdown-ссылки, которые Cursor открывает в редакторе: `[registry.py](/C:/Saby/src/crm/www/service/Модули/CRMGeneral/generals/crmclients/registry.py:1624)`.
- Перед тем как дать кликабельную ссылку, проверь что файл физически существует по этому абсолютному пути в текущем рабочем дереве.
- Если файл есть только в diff/MR, но отсутствует в текущем checkout, НЕ делай markdown-ссылку на `file` path. В таком случае пиши путь и строку обычным текстом и явно помечай, что файл есть только в MR/ветке и локально не открыт.
- Не используй цель ссылки вида `C:/...` без ведущего `/`: такая ссылка открывается как URL в браузере/webview, а не как файл в редакторе.
- Не оборачивай markdown-ссылку дополнительными квадратными или круглыми скобками. Надежный формат: `[label](/absolute/path:line)`.
